# RAG Summary Notebook

## 1. 什么是 RAG
RAG (Retrieval-Augmented Generation，检索增强生成) 是一种将信息检索与大语言模型生成能力相结合的方法范式。它并非仅依赖模型的参数化知识，而是先从外部知识源中检索相关信息，再基于这些证据生成有依据的响应。

RAG 之所以重要，在于它能够提升答案的事实一致性与可追溯性，并支持引入基础模型本身未覆盖的私有知识或垂直领域知识。在实际应用中，RAG 是降低幻觉风险、同时保持 LLM 在真实业务场景中可用性的最常见技术路径之一。

典型应用场景包括文档 Q&A、企业知识助手、研究型 copilot、客户支持，以及供应链、法务、金融、医疗等垂直领域的知识检索与问答场景。

## 2. 端到端工作流
### 2.1 高层流程
下面对 RAG 工作流的高层流程组成进行简要拆解：
- 数据接入与预处理：系统首先从文档、数据库、API 或其他知识源中采集原始内容，并进行清洗与标准化处理，以确保下游流程可以稳定使用这些数据。
- Chunking 与元数据增强：将内容切分为便于管理与检索的最小单元，并为每个 chunk 补充来源、日期、主题、实体、文档章节等有价值的元数据。
- 面向向量检索、关键词检索或混合检索的索引构建：将处理后的内容写入一个或多个检索索引，以支持后续基于语义、精确关键词，或两者结合的检索。
- Query 理解与检索：当用户发起问题时，系统会先理解 query 的意图，再从已建立索引的知识库中检索最相关的证据片段。
- 重排序与上下文组装：对检索结果进行过滤、重排与聚合，形成更聚焦、更相关、且适合模型消费的上下文包。
- 有依据的生成：语言模型基于检索得到的上下文生成答案，并尽量保证输出与支撑证据保持一致。
- 评估与迭代优化：通过指标评估、测试或人工反馈对最终输出进行审视，并据此持续优化数据、检索策略或提示词设计。

### 2.2 核心逻辑
- 更优的源数据会提升检索质量
  如果原始数据存在缺失、过时、噪声过多或结构不良等问题，检索器可利用的基础就会变弱，进而影响整体检索效果。
- 更优的检索会提升上下文质量
  检索决定了哪些证据能够进入模型的上下文窗口，因此检索能力薄弱通常会导致依据不完整或上下文不相关。
- 更优的上下文会提升答案质量
  当模型接收到相关性更高、组织更清晰的证据时，更有可能生成准确、可用且可追溯的回答。
- 评估会优化整个闭环
  评估有助于识别问题究竟来源于数据、检索、上下文构建还是生成阶段，从而支持系统性的优化迭代。

## 3. 核心概念
这些概念构成了理解 RAG 系统的基础术语体系。掌握这些基本要点后，不同 RAG 方法变体之间的差异也会更容易比较。

### 3.1 检索概念
本节覆盖决定 RAG 系统如何存储、搜索并返回相关证据的核心构件。

- Chunk：chunk 是从文档中切分出的更小文本单元，便于高效建立索引与执行检索。实际应用中，chunking 常基于固定 token 窗口、句子或段落边界，或依据标题与章节结构进行结构感知切分。常见工具包括 tokenizer、文档解析器，以及 LangChain、LlamaIndex 等框架中的 chunking utilities。
- Embedding：embedding 是用于表示 chunk 或 query 语义信息的稠密数值向量。通常先由 embedding model 生成，再写入向量数据库以支持相似度检索。常见工具包括 OpenAI embedding models、Sentence Transformers，以及 FAISS、Pinecone、Weaviate、Chroma、Milvus 等向量存储。
- 向量检索：向量检索通过比较 query embedding 与文档 embedding 之间的语义相似度来完成检索。这在用户采用自然语言提问、而相关知识源使用不同表达方式时尤其有效。典型实现依赖于 FAISS、Pinecone、Weaviate、Milvus 或 pgvector 中的近似最近邻索引。
- 关键词检索 / BM25 检索：关键词检索基于精确或近似精确的术语匹配进行检索，通常使用 BM25 或相关词法评分方法。它尤其适用于产品名称、编码、实体名称、稀有领域术语以及合规类查询。常见工具包括 Elasticsearch、OpenSearch、Solr 与 PostgreSQL 全文检索。
- 混合检索：混合检索将语义检索与关键词检索结合，使系统能够同时利用语义相似性与精确词项匹配能力。当 query 同时包含自然语言表达与领域标识符时，这通常是最强的生产级基线方案。常见实现方式包括在 OpenSearch、Elasticsearch、Weaviate 或自定义编排层中进行分数融合或并行检索。
- Top-k 检索：Top-k 检索指在重排序或提示词组装之前，从检索器返回结果中选取排名最高的 `k` 条。`k` 的取值控制着召回与噪声之间的权衡：更大的 `k` 可能带来更高的证据召回率，但也可能引入更多无关内容。该步骤通常直接在 LangChain、LlamaIndex、Haystack 或自定义流程中的检索器逻辑里配置。

#### 检索方式对比
| 检索方式 | 工作方式 | 最适用场景 | 典型工具 |
|---|---|---|---|
| 向量检索 | 在 embedding space 中基于语义相似度匹配 query 与文档 | 自然语言提问、同义改写 query、概念级检索 | FAISS, Pinecone, Weaviate, Milvus, pgvector |
| 关键词检索 / BM25 检索 | 基于精确词项匹配并计算词法相关性分数 | 产品编码、法务条款、模型名称、稀有术语、精确短语检索 | Elasticsearch, OpenSearch, Solr, PostgreSQL FTS |
| 混合检索 | 结合向量检索与关键词检索，再对结果进行融合或重加权 | 企业级 RAG、自然语言与标识符混合 query、生产级搜索 | OpenSearch, Elasticsearch, Weaviate, custom retrieval pipelines |

### 3.2 生成概念
本节关注检索到的证据如何被语言模型转化为最终答案。

- 提示词 grounding：提示词 grounding 指通过提示词设计引导模型基于检索到的证据作答，而不是仅依赖其参数化知识。这通常涉及明确的提示词指令、插入式上下文，以及在部分场景下的 citation 要求。
- 上下文窗口：上下文窗口指模型在单次处理过程中可接收的文本容量，其中包括用户 query、system instruction 以及检索到的证据。它直接影响可传入生成阶段的支撑材料规模。
- 引用 / 来源归因：citation 或 source attribution 用于将模型输出与支撑该回答的文档或 chunk 建立对应关系，从而提升答案的可追溯性、用户信任度以及在业务或合规场景中的可审查性。
- 幻觉 vs 有依据回答：幻觉指内容看似合理、但缺乏证据支持的回答；有依据回答则明确建立在检索到的上下文之上。RAG 的核心目标之一，就是推动生成结果尽可能从幻觉式回答转向有依据的回答。

### 3.3 系统层面概念
这些系统层面的概念有助于理解在设计与运行 RAG 方案时所面临的关键权衡。

- 召回率 vs 精确率：召回率衡量系统是否能够找回相关证据，精确率衡量返回结果中相关内容所占比例。一个鲁棒性高的 RAG 系统通常需要在两者之间取得良好平衡。
- 时延 vs 质量：更高质量往往来自额外步骤，例如 query rewriting、混合检索、重排序或迭代式推理，但这些步骤通常也会带来更高的响应时延。该权衡在交互式应用中尤为重要。
- 成本 vs 性能：更优的 embedding、更大的模型、更多级检索阶段以及重排序机制都可能提升输出质量，但也会增加算力与基础设施成本。生产系统通常追求的是可接受质量下的成本效率，而非单纯的最高质量。
- 静态流程 vs Agentic 流程：静态流程按固定的检索与生成流程运行，而 Agentic 流程则可以动态决定如何搜索、检索或调用工具。前者通常更简单、可预测性更强，后者则更灵活，但系统复杂度也更高。

## 4. 常见 RAG 方法
RAG 方法可以按照其主要优化目标进行分类，例如基础检索能力、检索质量、知识结构建模能力，或系统控制与可靠性能力。

### 4.1 基础检索方法
#### 朴素 / 基础 RAG
- 原理：该方法从知识库中检索最相关的 chunk，并将其直接作为支撑上下文传递给 LLM。
- 适用场景：它适用于简单文档问答、FAQ 助手，以及复杂度较低的早期 RAG 原型。
- 优势：它实现简单、成本较低，适合用于验证 RAG 流程的基础有效性。
- 局限：它高度依赖 chunk 质量和初始检索准确性，通常难以处理多跳推理或语义模糊的问题。
- 示例：例如，企业员工手册助手会检索最相关的制度条款 chunk，并据此回答员工问题。

#### 混合 RAG
- 原理：该方法将基于向量的语义检索与关键词或 BM25 检索结合，并将结果融合为更强的候选集合。
- 适用场景：它适用于企业知识搜索、技术文档检索，以及同时依赖自然语言和精确标识符的场景。
- 优势：它能够同时覆盖语义相似性与精确词项匹配，从而提升召回率，并且通常是较强的生产级基线方案。
- 局限：它会增加系统复杂度，需要设计分数融合或权重机制，且相比单一检索器可能带来更高时延。
- 示例：例如，供应链助手会对自然语言问题使用向量检索，同时对供应商名称、物料编码或零件号使用关键词检索。

### 4.2 检索增强方法
#### Reranking RAG
- 原理：该方法先检索较大的候选集合，再使用更强的相关性模型在生成前对结果进行重排序。
- 适用场景：它适用于初始检索结果噪声较大、系统需要提升最终上下文精确率的场景。
- 优势：它能够显著提升被选中证据的相关性，并减少传递给 LLM 的干扰性上下文。
- 局限：它会带来额外的推理成本与时延，并且其效果仍依赖于初始候选集合中是否已经包含正确证据。
- 示例：例如，法务助手先检索 30 条候选段落，再通过 cross-encoder 重排序，仅将前 5 条发送给模型。

#### Query Rewrite / 多 Query RAG
- 原理：该方法将一个 query 改写为更清晰的 query，或扩展为多个相关 query，以便检索器从不同语义角度执行搜索。
- 适用场景：它适用于语义模糊、信息不充分，或对话式表达较强的 query，因为原始表述往往不足以检索到充分证据。
- 优势：它能够提升召回率，缓解用户表述与源文档表述不一致的问题，并增强整体检索鲁棒性。
- 局限：它可能引入噪声较高或偏离目标的改写 query，同时也会增加检索次数与编排复杂度。
- 示例：例如，用户提出“为什么交付不稳定？”时，系统会将其扩展为关于发货延迟、供应商风险、库存短缺和物流中断的多个 query。

#### 父子 / Hierarchical RAG
- 原理：该方法先检索粒度更小的子级 chunk 以保证精确性，再回溯到其更大的父级段落或章节以补充完整上下文。
- 适用场景：它适用于长报告、制度手册、研究论文，以及任何结构化较强、单个 chunk 容易语义碎片化的文档。
- 优势：它能够在检索精度与上下文完整性之间取得平衡，从而减少答案建立在孤立片段上的风险。
- 局限：它需要更复杂的文档结构设计，并可能增加存储、索引以及提示词上下文长度的成本。
- 示例：例如，制度助手先检索到关于数据保留的句子级 chunk，再扩展到完整制度章节后进行回答。

### 4.3 结构感知方法
#### 图 RAG
- 原理：该方法将知识组织为实体及其关系，并沿着图结构中的连接进行检索，而不只是依赖文本相似度。
- 适用场景：它适用于关系密集型领域，例如供应链、生物医药、欺诈分析以及企业知识图谱场景。
- 优势：它能够支持多跳推理，显式表达结构化关系，并在复杂知识任务中提升可解释性。
- 局限：它的构建与维护成本较高，而且图谱抽取质量会显著影响下游效果。
- 示例：例如，供应商风险系统会连接供应商、地区、港口、物料与事件，以回答间接中断链路相关问题。

#### 多模态 RAG
- 原理：该方法能够跨文本、表格、图表、图像或 PDF 等多模态内容检索证据，并由多模态模型进行联合理解。
- 适用场景：它适用于关键信息分散在图像、表格、图表等结构化或视觉化载体中，而非仅存在于纯文本中的场景。
- 优势：它扩展了系统可利用的证据空间，并提升对混合格式真实文档的处理能力。
- 局限：它需要更复杂的解析、索引与评估流程，且基础设施成本通常高于纯文本 RAG。
- 示例：例如，金融助手会同时检索财报中的管理层文字说明与图表图像，以回答业绩表现相关问题。

### 4.4 控制与可靠性方法
#### Agentic RAG
- 原理：该方法允许 agent 自主决定如何拆解任务、选择哪些工具或检索器，以及是否需要追加新的检索步骤。
- 适用场景：它适用于复杂、多步骤，或依赖工具协同完成的任务，因为这类任务通常无法通过单次固定检索良好解决。
- 优势：它具备较高灵活性与自适应能力，能够处理跨多源数据与多工具协同的复杂推理流程。
- 局限：它更难治理、调试与评估，且通常会带来更高时延、成本以及行为不确定性。
- 示例：例如，研究型助手会先规划子问题，再查询多个知识源、执行计算，最后综合生成最终答案。
- 我的理解：

#### Self-RAG / Corrective RAG
- 原理：该方法允许系统判断当前检索证据是否充分，并在必要时触发追加检索或答案校正。
- 适用场景：它适用于高风险或高可靠性要求场景，因为在这些场景中，缺乏证据支撑的回答代价往往更高。
- 优势：它能够减少缺乏依据的生成内容，提升答案的依据性，并在流程内部形成更稳健的反馈机制。
- 局限：它会增加编排复杂度和额外的模型调用次数，同时其效果高度依赖自检逻辑本身的质量。
- 示例：例如，合规助手发现第一次回答缺乏证据后，会触发第二轮检索，并在返回前修正最终响应。

### 4.5 建议分类总结
- 核心检索：这些方法定义了将外部知识接入生成过程的基础检索模式。
- 检索增强：这些方法在基础检索流程建立后，进一步提升召回率、精确率或上下文完整性。
- 知识结构扩展：这些方法通过引入更丰富的知识结构或额外模态，使 RAG 超越单纯的文本相似度检索。
- 推理 / 控制扩展：这些方法增强了系统在规划、自我修正以及动态调整检索行为方面的能力。

## 5. 设计选择
本节聚焦于那些最直接影响检索质量、答案质量、响应时延与系统可维护性的关键架构选择。

### 5.1 数据设计
- 纳入哪些数据源：需要明确系统接入的是内部文档、数据库、API、外部报告，还是混合型知识源，因为数据源范围直接决定了系统可回答问题的边界。
- 数据质量与时效性：如果源数据存在错误、缺失、重复或过时问题，后续检索与生成质量通常也会同步受损。
- 元数据可用性：高质量元数据有助于支持过滤、路由、权限控制与结果解释，是提升检索精度的重要基础。
- 结构化数据 vs 非结构化数据：结构化数据更适合精确查询与字段过滤，非结构化数据更适合语义检索与知识补充，实际系统通常需要同时处理两者。

### 5.2 Chunking 设计
- 固定长度 Chunking：按照固定 token 或字符长度切分文档，优点是实现简单、便于批量处理，但容易切断语义边界。
- 语义 Chunking：按照句子、段落、标题或语义单元切分内容，通常更有利于保持上下文完整性，但实现复杂度更高。
- 重叠大小：相邻 chunk 之间保留一定重叠区域，有助于减少关键信息被边界截断的风险，但也会增加冗余与存储成本。
- Parent-Child Chunking：同时保留细粒度 chunk 与上层章节结构，便于在高精度检索后补充更完整的上下文。
- Chunk 大小权衡：较小的 chunk 往往有利于提升检索精度，较大的 chunk 则更有利于保留完整上下文，因此需要根据任务场景做平衡。
- 备注：

### 5.3 检索设计
- 密集检索：基于 embedding 的语义相似度进行检索，适合自然语言提问、概念匹配与表述差异较大的场景。
- 稀疏检索：基于关键词或 BM25 等词项匹配方式进行检索，适合精确术语、编码、专有名词或法规条款类查询。
- 混合检索：结合密集检索与稀疏检索，通常能够兼顾语义理解与精确匹配，是较常见的生产级基线方案。
- 元数据过滤：通过时间、来源、地区、文档类型、权限等元数据条件过滤候选结果，有助于提升检索相关性与治理能力。
- Top-k 选择：需要决定在初始检索阶段保留多少条候选结果，因为过小的 k 会损失召回，过大的 k 则容易引入噪声。
- 多阶段检索：先用轻量检索器召回候选集合，再通过重排序或专用策略做进一步筛选，适合对质量要求较高的系统。
- 备注：

### 5.4 Query 处理设计
- Query rewrite：将原始 query 改写为更清晰、更适合检索的表达，以缓解用户提问与知识源表述之间的不一致。
- Query 扩展：围绕同一问题补充同义词、相关术语或扩展条件，以提高召回率。
- Query 拆解：将复杂问题拆解为多个子问题，再分别检索相关证据，适合多步骤或多跳问题。
- 意图分类：先识别 query 属于事实查询、摘要请求、比较分析还是决策支持，再选择更合适的检索与生成策略。
- 检索器路由：根据 query 类型、数据源类型或任务复杂度，将请求发送到最合适的检索器或知识索引。
- 备注：

### 5.5 排序与上下文设计
- 是否使用 Reranking：需要判断系统是否需要在初始召回后再做重排序，以进一步提升最终上下文的相关性。
- 去重：对语义重复或来源重复的结果进行清理，可减少上下文浪费并降低模型受到重复信息干扰的风险。
- 上下文压缩：在上下文窗口有限的情况下，对检索结果进行摘要、截断或重点提取，以保留最有价值的信息。
- 引用封装：在上下文中保留来源标记、段落定位或文档编号，有助于提升输出的可追溯性与可审查性。
- 提示词结构：需要设计好系统指令、用户问题、检索证据与输出要求之间的组织方式，以确保模型稳定地基于证据作答。
- 备注：

### 5.6 系统权衡
- 召回率 vs 精确率：更高召回率有助于减少漏检，但往往也会引入更多噪声，因此需要平衡找得全与找得准。
- 质量 vs 时延：更复杂的 query 处理、检索增强与重排序流程通常能提升质量，但也会拉长整体响应时间。
- 简单性 vs 复杂性：简单架构更易实现、调试与维护，而复杂架构虽然能力更强，但治理与评估成本也更高。
- 成本 vs 可靠性：更多模型调用、更高质量 embedding 与更复杂的流程往往能提升可靠性，但也会显著增加系统成本。
- 备注：

### 5.7 建议的默认基线
- 干净的源数据：应优先保证数据接入质量，因为这是后续检索与生成效果的基础。
- 高质量元数据：建议尽早设计好来源、时间、主题、实体等元数据，以支持过滤、治理与解释。
- 混合检索：对于多数企业场景，混合检索通常比单一检索方式更稳健。
- Reranking：当检索结果噪声较高时，引入重排序通常是最直接的质量提升手段之一。
- 文档较长时使用 Parent-Child 上下文：对于长文档或层级结构明显的内容，应优先考虑通过层级结构补充上下文完整性。
- 在引入 Agentic 复杂性前先做好评估：在系统基础能力尚未被充分验证前，不宜过早引入更复杂的 agentic 编排机制。

当这些设计选择明确之后，下一步就需要通过系统化评估来验证它们是否真正有效。下一节将聚焦于如何衡量 RAG 系统的实际表现。

## 6. 评估指标
评估是区分一个“看起来可行”的原型与一个“真正可靠”的系统的关键环节。它有助于判断问题究竟来自检索、排序、提示词设计，还是生成阶段。

### 6.1 检索指标
- 召回率 / 命中率
  检索器是否找到了真正相关的证据？
- 精确率
  检索结果中真正有用的内容占比有多高？
- MRR / NDCG
  相关结果在排序中的位置是否足够靠前？
- 过滤准确率
  元数据过滤条件是否正确筛选出了目标结果子集？
- 备注：

### 6.2 生成指标
- 事实一致性 / 依据性
  答案是否真正建立在已检索证据之上？
- 相关性
  回答是否切中了用户提出的问题？
- 完整性
  是否遗漏了关键事实或必要信息？
- 简洁性 / 清晰度
  回答是否清晰、易用且不过度冗长？
- 备注：

### 6.3 端到端产品指标
- 任务成功率
- 用户满意度
- 响应时延
- 单次 query 成本
- 引用有效性 / 可追溯性
- 备注：

### 6.4 评估方法
- 离线基准测试集
- 人工评审
- Golden Q&A 集
- A/B 测试
- 生产环境监控
- 备注：

### 6.5 重要评估原则
- 将检索与生成分开评估
- 否则很难判断问题究竟来自数据、搜索、排序还是提示词设计

当评估标准明确之后，就更容易识别 RAG 系统常见的失效位置，以及实施过程中需要重点关注的风险点。

## 7. 风险与失效模式
RAG 系统几乎可能在每一层出现失效，从源数据到最终答案生成都存在潜在风险。将这些风险显式列出，有助于把技术设计选择与实际业务后果更紧密地联系起来。

### 7.1 数据风险
- 源数据质量差：如果原始数据本身存在错误、缺失、噪声过多或结构混乱，那么后续检索与生成结果往往会在错误基础上继续放大问题。
- 数据过时：即使历史数据曾经准确，如果没有及时更新，也可能导致系统给出与当前事实不一致的回答。
- 重复数据：相同或高度相似的数据多次出现，容易让检索阶段重复召回相似证据，进而干扰排序与上下文构建。
- 合规 / 隐私问题：如果数据中包含敏感信息、受限内容或未授权使用的数据源，系统可能在生成过程中暴露隐私或触发合规风险。

### 7.2 检索风险
- 漏掉相关证据：如果检索器未能召回关键证据，即使后续模型能力很强，也可能因为“证据缺席”而给出错误答案。
- 噪声证据过多：当候选结果中包含大量无关内容时，模型更容易受到干扰，导致答案偏题或依据不集中。
- 元数据过滤薄弱：如果时间、来源、权限、地区等过滤条件设计不充分，系统可能会检索到不应被使用的内容或遗漏更合适的结果。
- Chunking 策略不当：chunk 切分过细可能导致语义碎片化，切分过大又可能降低检索精度，因此错误的 chunking 策略会直接损害检索质量。

### 7.3 生成风险
- 幻觉：模型可能生成看似合理、但并未真正得到证据支持的内容，这是 RAG 系统最典型的风险之一。
- 缺乏依据的综合：即使模型使用了部分正确证据，也可能在综合多个片段时引入原文并未明确支持的推断。
- 语气过度自信：当模型以非常确定的方式表达不完全可靠的结论时，会显著放大业务使用风险。
- 引用丢失：如果生成阶段未正确保留来源标记或引用关系，即使答案本身接近正确，也会削弱其可追溯性与可审查性。

### 7.4 系统风险
- 时延过高：过多的检索、重排序、工具调用或多轮推理会使整体响应时间过长，从而影响实际可用性。
- 成本过高：高频模型调用、大规模索引维护、多阶段检索与复杂编排都可能使系统运行成本快速上升。
- 调试困难：当系统包含多个检索器、重排序器、提示词模板与 agentic 流程时，定位问题来源会变得更加困难。
- 评估盲区：如果缺乏系统化评估，团队可能误以为系统表现良好，但实际上某些关键失败模式尚未被发现。
- 过早过度设计：在基础能力尚未验证前就引入过于复杂的架构，往往会增加系统负担，而不一定带来与之匹配的价值提升。

这些风险也解释了为什么成熟团队通常会让 RAG 系统逐步演进，而不是一开始就直接采用最复杂的架构设计。

- 仅在确有必要时引入 Agentic 工作流

### 8.3 不同业务场景下的可选设计
- 通用企业知识库场景：如果数据主要来自制度文档、知识库页面、FAQ 和内部说明材料，通常可从 `混合检索 + Reranking + 强元数据 + grounded prompting` 起步，因为这类场景既需要语义理解，也需要精确术语匹配。
- 法务、合规、审计场景：如果数据强调条款精确性、来源可追溯性与低幻觉风险，通常更适合采用 `关键词 / BM25 检索 + 混合检索 + Reranking + citation 强约束 + Self-RAG / Corrective RAG`。
- 供应链、风控、关系分析场景：如果数据中存在大量实体关系，例如供应商、地区、物料、港口、事件或依赖链路，通常更适合采用 `混合检索 + Graph RAG + 元数据过滤 + 多跳推理支持`。
- 研发文档、技术手册、长报告场景：如果输入以长篇结构化文档为主，且答案依赖上下文完整性，通常更适合采用 `Hierarchical RAG / Parent-Child RAG + 混合检索 + Reranking`。
- 多模态文档场景：如果关键信息分布在文本、表格、图表、图片或 PDF 页面中，通常应优先考虑 `多模态 RAG + 版面解析 + 结构化抽取 + 多模态模型生成`。
- 高度结构化数据场景：如果输入主要是表格、数据库记录、特征字段或指标流，而非大段文本，通常可采用 `字段级检索 + 元数据过滤 + 混合检索`，必要时再结合规则引擎或 SQL 查询能力。
- 对话式助手或模糊 query 较多的场景：如果用户问题经常表达不完整、口语化或多轮变化明显，通常更适合采用 `Query Rewrite / 多 Query RAG + 混合检索 + Reranking`。
- 高复杂度研究或分析场景：如果任务需要跨多个知识源检索、拆解子问题、执行多步骤推理或工具调用，通常更适合采用 `Agentic RAG + 多检索器路由 + 工具调用 + 评估闭环`。
- 数据更新频繁的场景：如果数据时效性要求高，例如市场动态、库存状态、舆情、运营指标或实时告警，应重点强化 `增量索引更新 + 强时间元数据 + 检索过滤 + 评估监控`，而不是只关注生成层优化。
- 冷启动或早期 PoC 场景：如果目标是尽快验证价值，建议从 `朴素 RAG` 或 `混合检索 + 基础元数据` 开始，再根据评估结果逐步引入 Reranking、层级结构或 Agentic 能力。
