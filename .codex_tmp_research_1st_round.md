# RAG 第一轮研究

## 1. 研究范围
这份笔记用于记录我基于 `all-in-rag` 学习仓库开展的第一轮 RAG 研究，重点是理解标准 RAG 流程背后的基础概念，以及各阶段常见的工具或模型。

这一轮研究的范围偏入门与系统梳理，主要用于整理我对以下内容的理解：
- RAG 核心概念
- 从文本解析到检索、召回与生成的高层工作流
- 实践中常见的工具、库、向量库和嵌入模型
- 不同实现选择之间的基础对比

这一轮研究并不以产出最终最佳实践架构为目标，而是作为一份用于基础学习、对比和后续扩展的结构化研究笔记。

### 4.2 分块策略

#### 基础实验背景
- 这是一个在相同分块策略下，只调整 `chunk_size` 和 `chunk_overlap` 的基础实验。
- 实验基于 `all-in-rag` Chapter 1。
- 被分析文档为 `all-in-rag/data/C1/early_rl_chapter1.md`。
- 目标是观察：在检索链路和其余条件不变时，分块超参数如何影响分块数量、上下文 token 开销、缓存命中率，以及最终回答完整度。

#### 实验结果
| 测试 | 参数 | Chunks | Prompt tokens | Cache hit rate | 实际输出结果 |
|---|---|---:|---:|---:|---|
| 1 | `500 / 50` | 61 | 1074 | `95.34%` | 返回了若干强化学习应用例子，但明确表示无法回答“人类探索与利用行为”的具体例子 |
| 2 | `1000 / 100` | 26 | 2494 | `97.51%` | 比测试 1 更完整，补充了更多强化学习例子，但仍未回答具体的人类行为例子 |
| 3 | `2000 / 150` | 13 | 4843 | `97.79%` | 两部分都能完整回答，包含餐馆、广告、挖油、游戏等例子，整体较均衡 |
| 4 | `4000 / 200` | 6 | 9473 | `99.99%` | 整体答案最完整，既包含较全面的强化学习应用，也覆盖全部人类探索与利用例子 |
| 5 | `8000 / 400` | 3 | 11422 | `99.74%` | 两部分都能答对，但强化学习应用列表开始变短，说明检索粒度已经变粗 |

#### 结果解读
- 当 `chunk_size` 较小时，虽然分块更细、检索粒度更高，但相关信息容易被拆散，导致回答只能覆盖部分子问题。
- 从 `500 / 50` 到 `1000 / 100`，回答完整度有提升，但仍不足以覆盖文中后半部分的人类行为例子。
- `2000 / 150` 是一个明显的拐点：此时模型已经能够同时覆盖两个子问题，说明单个 chunk 中保留了更完整的上下文。
- `4000 / 200` 给出了质量最好的答案，但对应的上下文 token 成本也明显更高。
- 当参数进一步放大到 `8000 / 400` 时，虽然答案仍然正确，但回答开始变得更“粗”，说明过大的 chunk 会让检索变得不够精细。

#### 简短结论
- 在这组实验里，`2000 / 150` 仍然是最均衡的默认选择。
- 如果优先追求回答质量，`4000 / 200` 是更好的设置。
- `8000 / 400` 仍然可用，但已经能观察到检索粒度变粗、回答覆盖面下降的问题。

#### 按场景的分块超参数建议
| 场景 | 建议参数方向 | 说明 |
|---|---|---|
| 需要一个通用、稳妥的默认值 | `2000 / 150` 左右 | 在回答完整度与上下文成本之间更均衡，适合作为基础 baseline |
| 优先追求回答质量与上下文完整性 | `4000 / 200` 左右 | 更容易保留跨段信息，但 token 成本更高 |
| 文档本身较短、问题常常需要跨段综合 | 偏大 `chunk_size` | 可以减少信息被切散的风险 |
| 文档很长、主题切换频繁、希望检索更精细 | 偏小 `chunk_size` | 有助于提升定位精度，但要警惕上下文被切碎 |
| 希望减少重复但保留一定衔接 | 中等 `chunk_overlap` | overlap 太小会丢衔接，太大则增加冗余和 token 成本 |
| 已经出现“能答对但不够全面”的情况 | 适当增大 `chunk_size` | 说明每个 chunk 承载的信息可能还不够完整 |
| 已经出现“召回内容太大、答案变粗”的情况 | 适当减小 `chunk_size` | 说明检索粒度可能过粗，需要提高定位能力 |

总结：
- `chunk_size` 决定信息是被切得更细，还是被保留得更完整。
- `chunk_overlap` 主要用于补足相邻 chunk 之间的衔接，但不是越大越好。
- 选参数时不要只看是否“答对”，还要看回答是否完整、检索是否过粗，以及 token 成本是否可接受。

### 4.3 嵌入模型

#### 使用策略 （脑爆）
- 不同上下文应使用不同的 API 调用方式，不建议所有输入都走同一套嵌入流程。
- 可以先识别输入文档类型，再决定使用哪一类嵌入；或者在后端按数据类型拆分 DB，并通过标签区分检索来源。
- 一个简单原则是：只有在内容本身确实是混合内容时，才优先使用多模态嵌入。

#### 按场景的模型建议
| 数据类型 / 场景 | 建议模型选择 |
|---|---|
| 以简体中文纯文本、Markdown、PDF 抽取文本为主 | **`BAAI/bge-m3`**、**`Qwen/Qwen3-Embedding-8B`**，或 Cohere / Voyage 的文本嵌入模型 |
| 中文 PDF 中包含表格、图表、截图，且版面信息本身有意义 | **`voyage-multimodal-3`** 或 **`embed-v4.0`** |
| 中文扫描文档，且 OCR 可能损失结构信息的情况 | **`voyage-multimodal-3`**、**`Qwen/Qwen3-VL-Embedding-8B`**，或 **`bge-visualized-m3`** |
| 中文图片检索、流程图、商品图片、截图 | **`jina-clip-v2`**、**`Qwen/Qwen3-VL-Embedding-8B`**，或 **`bge-visualized-m3`** |
| 需要一个模型同时覆盖 纯文本、图像、音频、视频、PDF | **`gemini-embedding-2`** |
| 需要本地 / 私有化部署，面向中文企业文档 | **`Qwen/Qwen3-VL-Embedding-8B`** 或 **`BAAI/bge-visualized`** |
| 需要一个简单的中文 RAG 基模 | `OCR / PDF parser → BAAI/bge-m3 → reranker` |

总结：
- 如果数据以中文文本为主，优先从文本嵌入模型开始，通常更稳妥。
- 如果文档的版面、图表、截图本身承载语义，多模态模型更合适。
- 如果需要兼顾私有化部署与中文能力，Qwen 和 BAAI 系列更值得优先评估。
- 如果只是先搭一个中文 RAG baseline，`OCR / PDF parser → BAAI/bge-m3 → reranker` 是一条简单且实用的起点。

#### 关于同一嵌入模型的不同嵌入模态的简单实验记录
- 本实验围绕 `Datawhale` 开源组织 logo 的相似度表现展开。
- 对象为两个相似但不完全相同的 logo 版本：
  - 图片 1：带有文字 `Datawhale`
  - 图片 2：不带文字
- 对比了 4 组相似度：
  - 第 1 组：纯图像 vs 纯图像
  - 第 2 组：图文结合 1 vs 纯图像
  - 第 3 组：图文结合 1 vs 纯文本
  - 第 4 组：图文结合 1 vs 图文结合 2
- 其中，“图文结合”表示该输入使用 多模态 嵌入。

#### 实验结果
| 文本描述 | 第 1 组：纯图像 vs 纯图像 | 第 2 组：图文结合 1 vs 纯图像 | 第 3 组：图文结合 1 vs 纯文本 | 第 4 组：图文结合 1 vs 图文结合 2 | 最高相似度组别 |
|---|---:|---:|---:|---:|---|
| `datawhale开源组织的logo` | 0.8318 | 0.8291 | 0.7627 | 0.9058 | 第 4 组 |
| `blue whale` | 0.8318 | 0.9218 | 0.7572 | 0.8719 | 第 2 组 |
| `whale` | 0.8318 | 0.9713 | 0.7255 | 0.8648 | 第 2 组 |

#### 结果解读
- 当文本能够非常准确地描述图像时，两个多模态嵌入向量之间的相似度最高。
- 当文本只抓住图像的主要特征、但描述不够具体时，最高相似度反而出现在“图文结合 vs 纯图像”之间。
- 随着文本描述变得更抽象，从 `blue whale` 到 `whale`，第 2 组的相似度进一步升高，说明嵌入更容易向图像的主视觉特征收敛，而不是保留更完整的语义约束。

#### 简单结论
- 嵌入 model 的使用应当按输入类型定制，而不是统一处理所有内容。
- 多模态 嵌入 更适合真正的混合内容输入，不应默认替代所有单模态场景。
- 当输入格式不同，例如 image、text、OCR 处理后的文本，即使它们实际表达的是相近内容，两个多模态嵌入向量的相似度也可能下降。
- 如果文本描述足够清晰、足够具体，多模态嵌入往往能更好地保留图文联合语义，此时相似度表现最好。

#### 风险与启发
- 三组试验中，相似度最高值=0.9713，此试验疑似仅提取了图片的鲸鱼特征而丢失颜色，含义等信息。
- 如果文本描述过于抽象，嵌入 结果可能更接近“纯图像特征”，从而意外召回一些视觉上相关、但语义上并不真正相似的内容。
- 例如，`whale` 这样的描述可能让 Datawhale logo 与真实鲸鱼图片之间出现不符合预期的相关性。
- 同时，过于抽象的文本还可能削弱输入中的关键领域信息，例如 Datawhale 作为“开源组织”这一身份特征，进而降低它与其他“开源组织”相关内容之间本应较高的相似度。
- 因此，在系统设计上，更合理的做法是按内容类型选择嵌入路径，并在索引层或检索层保留类型标签与来源标签，避免不同格式内容被过度混合。
